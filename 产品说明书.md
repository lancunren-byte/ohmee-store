## 一、产品概述

### 1.1 产品背景

Ohmee 排班打卡系统专为便利店连锁门店设计，解决多门店场景下人员排班复杂、打卡管理分散、数据汇总困难等痛点。系统覆盖从基础数据配置、人员排班、员工打卡到数据看板的完整闭环，支持直营店、托管店、特许加盟店等多种门店形态。

### 1.2 产品定位

| 维度 | 说明 |
|------|------|
| 目标用户 | 便利店连锁总部运营团队、区域督导、门店店长、一线店员 |
| 使用场景 | 多门店排班管理、日常上下班打卡、人员跨店借调协作 |
| 技术平台 | PC 浏览器端 + 手机移动端（H5，支持微信/浏览器打开） |

### 1.3 核心价值

- **规范排班**：统一班次管理，排班记录在线可查，告别纸质表格
- **精准打卡**：GPS 定位 + 人脸拍照双重验证，防止代打卡、虚假打卡
- **灵活借调**：跨店借调线上流转，调入门店发起 → 调出门店审批，权责清晰
- **数据看板**：实时汇总考勤数据，迟到、早退、缺勤一目了然

### 1.4 多语言支持

系统支持 **简体中文（中文）/ English（英文）/ Tiếng Việt（越南语）** 三种界面语言，满足不同国籍员工的使用需求。

### 1.5 时区自适应

系统支持**多国家、多时区**部署，所有时间展示与业务逻辑均按用户所在时区自动适配，确保中国（UTC+8）、越南（UTC+7）等不同地区用户看到的时间一致、准确。

#### 时区确定规则

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | 当前所选国家 | PC 端顶部选择国家后，采用该国标准时区（如中国 UTC+8、越南 UTC+7） |
| 2 | 用户所属公司所在国家 | 移动端及未切换国家时，按用户 `companyId` 对应国家的时区 |
| 3 | 浏览器/设备时区 | 无法从业务数据推断时，回退为设备本地时区 |

#### 覆盖范围

| 场景 | 时区处理 |
|------|----------|
| **打卡时间** | 上班/下班打卡记录以**门店所在国家时区**存储与展示 |
| **班次时间** | 班次开始/结束时间为门店当地时间，跨零点班次按门店日期边界计算 |
| **今日/本月统计** | 「今日打卡」「本月出勤」等以**当前用户时区**的 0 点与月末为边界 |
| **日期筛选器** | 日期选择器、快捷按钮（今日/本周/本月）按用户时区解释 |
| **列表与导出** | 所有时间字段（创建时间、修改时间、打卡时间）统一格式化为用户时区显示 |

#### 存储与展示规则

- **存储**：后端统一以 UTC 或门店所属国家时区存储，便于跨时区比对
- **展示**：前端根据当前用户时区将时间转换为本地可读格式（如 `YYYY-MM-DD HH:mm`）
- **跨时区场景**：督导查看多国门店时，各门店数据按各自国家时区展示，避免混淆

### 2.3 权限管理设计

权限分为三类：**数据权限**（基于国家、公司、区域、门店）、**菜单权限**（基于角色）、**按钮权限**（基于角色）。

#### 2.3.1 数据权限

| 角色 | 数据范围 | 说明 |
|------|----------|------|
| 督导 | 区域 | 仅可访问所辖门店（supervisorId = 当前用户）的数据 |
| 稽核专员 | 公司 | 可访问公司内全部区域、门店、员工数据 |
| 店长 | 门店 | 仅可访问本店数据 |
| 全职店员/管培生/兼职店员 | 本人 | 仅可访问与本人相关的打卡、排班等数据 |

数据权限按国家 → 公司 → 区域 → 门店逐级过滤。PC 端顶部国家/公司选择器会根据当前用户角色限制可选范围；督导、店长只能选择其所属公司；稽核专员可选择公司内任意公司。

#### 2.3.2 菜单权限

| 角色 | PC 端可见菜单 |

| 督导 | 系统概览、门店管理、员工管理、班次类型配置、交接班管理、**权限管理** |
| 稽核专员 | 同上 |
| 店长 | 系统概览、交接班管理 |
| 店员 | 系统概览 |

无权限的菜单项不显示；直接访问无权限路由时自动跳转至系统概览。

### 2.4 督导专属功能说明

督导角色负责多个门店的运营监管，在移动端具备以下专属能力：

#### 2.4.1 门店切换器

督导登录后，各主要页面（打卡看板、排班管理、借调管理）顶部均显示**门店切换器**，展示当前选中门店名称，点击可弹出门店列表，快速切换至所辖任意门店视角。

- 切换门店后，当前页面数据刷新为所选门店的数据

| 操作 | 督导 | 店长 |

| 新建/编辑/删除班次 | ✓ | ✓ |
| 查看人员排班列表 | ✓ | ✓ |

#### 3.1.2 考勤分析（默认页签）

**今日打卡情况（按班次类型分组）：**

以卡片形式展示当日各班次类型的出勤统计：

| 列名 | 说明 |
|------|------|

**异常数排名：**

| 列名 | 说明 |

| 门店名称 / 督导姓名 | — |
| 所属区域 | — |
| 异常总数 | 迟到 + 早退 + 缺勤之和 |
| 迟到次数 | — |
| 早退次数 | — |
| 缺勤次数 | — |
| 趋势 | 与上月对比，↑ 上升（变差）/ ↓ 下降（改善）|

> 可在「前 10」和「后 10」之间切换；门店维度和督导维度通过页签切换。

#### 3.1.3 任务分析（页签二）

| 指标 | 说明 |

### 3.2 区域管理

**功能入口**：门店管理 → 区域管理

区域是门店和员工的归属维度，用于支撑督导管理和数据分组。系统支持**多级树状区域结构**，可根据公司实际组织层级灵活配置，例如：大区 → 省区 → 城市区域，或 战区 → 片区 → 商圈 等多种层次划分。

**树状结构示例：**

```

        ├── 徐汇商圈

    ▼ 江苏片区
        ├── 南京区域
        └── 苏州区域
▼ 华南战区
    ▼ 广东片区
        ├── 广州天河区域
        └── 深圳南山区域
```

#### 操作说明

| 操作 | 说明 |
|------|------|
| 新增顶级区域 | 不选择父级区域，直接创建一级区域（如：华东战区） |

#### 关于"所属门店"的重要说明

**所属门店是员工的组织属性，代表员工的编制归属，与临时借调无关，任何情况下均不应随借调操作而变更。**

| 场景 | 所属门店 | 打卡验证门店 | 说明 |
|------|:--------:|:------------:|------|
| 正常上班 | 原店（组织归属） | 原店 | 所属门店 = 打卡验证门店 |
| 借调期间 | 原店（不变） | 借调目标门店 | 系统自动取当日有效借调记录中的门店经纬度进行验证 |
| 借调结束后 | 原店（不变） | 原店 | 借调失效，自动回归原店打卡 |

> **打卡验证逻辑**：员工每次打卡时，系统优先查询当天是否存在状态为"已批准"且在有效期内的借调记录。若存在，则以**借调目标门店**的经纬度作为打卡距离验证基准；否则，以**所属门店**的经纬度为基准。员工档案中的所属门店字段全程不受影响。

> **调店（永久）与借调（临时）的区别**：若员工永久调至新门店，应由管理员在员工档案中修改"所属门店"；若只是临时支援，则使用借调功能，所属门店保持不变。

### 3.8 工时统计

**功能入口**：PC 端侧边栏 → 数据报表 → 工时统计

为管理员提供各门店员工工时汇总查询，支持快捷日期选择和自定义区间，帮助核算薪资、评估出勤质量。

#### 工时计算规则

系统根据每条打卡记录的实际状态，按以下逻辑计算有效工时：

| 打卡状态 | 计算方式 | 说明 |
|----------|----------|------|
| **正常**（上班准时，下班准时）| 班次结束时间 − 班次开始时间 | 按排班计划全额计算 |
| **迟到**（上班迟到，下班正常）| 班次结束时间 − 实际上班打卡时间 | 迟到部分不计入工时 |
| **早退**（上班正常，下班早退）| 实际下班打卡时间 − 班次开始时间 | 早退部分不计入工时 |
| **迟到 + 早退** | 实际下班打卡时间 − 实际上班打卡时间 | 两端均扣减 |
| **缺勤**（未打卡）| 0 小时 | 不计入有效工时 |

> **注意**：跨零点班次（如 23:00—07:00）自动处理为 8 小时，不受日期边界影响。

#### 页面结构

**筛选条件：**

| 筛选项 | 类型 | 说明 |
|--------|------|------|
| 快捷选择 | 按钮组 | 本月 / 本周（点击即刻加载数据）|
| 自定义日期范围 | 日期区间选择器 | 可任意设定开始/结束日期 |
| 所属门店 | 下拉单选 | 不选则汇总全部门店 |
| 员工姓名/工号 | 文本搜索 | 模糊匹配 |
| 角色 | 下拉多选 | 按角色筛选 |

**汇总指标（顶部统计卡）：**

| 指标 | 说明 |
|------|------|
| 总计划工时 | 所选范围内所有员工排班计划工时总和 |
| 总实际工时 | 按规则计算的实际有效工时总和 |
| 工时达成率 | 实际工时 / 计划工时，百分比显示 |
| 缺勤总次数 | 所选范围内未打卡的排班次数 |

**数据表格（按员工汇总）：**

| 列名 | 说明 |
|------|------|
| 员工工号 | — |
| 员工姓名 | — |
| 所属门店 | — |
| 排班天数 | 所选时间段内的排班总天数 |
| 实出勤天数 | 有打卡记录（不含缺勤）的天数 |
| 迟到次数 | — |
| 早退次数 | — |
| 缺勤次数 | — |
| 计划工时（h）| 按班次时长累计 |
| 实际工时（h）| 按上述规则计算后累计，保留一位小数 |
| 工时差（h）| 实际工时 − 计划工时，负数红色显示 |

支持导出当前筛选结果为 Excel，文件名格式：`工时统计_YYYYMMDD_YYYYMMDD.xlsx`。

### 3.10 门店任务管理

**功能说明**：允许总部/运营团队配置标准化门店执行任务（商品陈列、清洁卫生等），按照设定周期和范围自动下发给对应门店，门店员工在移动端完成拍照反馈，督导可对任务执行情况打分审核。

#### 3.10.1 任务配置

**功能入口**：门店任务管理 → 任务配置

用于创建和维护可复用的任务模板，配置完成后系统按周期自动下发。

**任务基本信息：**

| 字段 | 类型 | 说明 |
|------|------|------|
| 任务名称 | 文本 | 必填，如"商品陈列标准检查" |
| 任务类型 | 下拉单选 | 商品陈列 / 熟食制作 / 清洁卫生 / 员工形象 |
| 是否需要审批 | 开关 | 是：门店提交后，督导需打分审核；否：提交即完成 |
| 任务状态 | 开关 | 启用 / 停用 |

**任务执行周期：**

| 周期类型 | 配置方式 | 示例 |
|----------|----------|------|
| 按时间段 | 起止日期 + 每日时间段 | 2026-03-01 至 2026-06-30，每日 10:00 前 |
| 每逢周X | 勾选星期几 + 截止时间 | 每周一、三、五，23:00 前完成 |
| 每月XX号 | 选择日期 + 截止时间 | 每月 5 日 18:00 前完成 |

**任务下发范围：**

| 选项 | 说明 |
|------|------|
| 全部门店 | 系统内所有营业中门店 |
| 按区域 | 选择一个或多个区域，自动包含该区域下所有门店 |
| 按门店 | 手动勾选具体门店 |
| 导入门店清单 | 上传 Excel 文件，批量指定门店 |

**下发与反馈时间：**

| 字段 | 说明 |
|------|------|
| 任务下发时间 | 系统自动下发任务的时间点（如每次周期开始时）|
| 任务反馈截止时间 | 门店必须完成并提交的截止时间 |

**任务内容（支持多条）：**

每条任务内容为一个执行项，包含：
- **执行标准文字**：描述该项的完成标准（如"货架陈列按照标准图示摆放，无缺货"）
- **标准参考图片**：可上传多张标准图片供员工参考
- **任务拍照要求**：员工在移动端**只允许拍照，不允许从相册上传**，拍照时自动记录时间和 GPS 位置

> **自动下发逻辑**：系统按配置时间自动将任务下发给对应门店。优先下发给当前有班的**店长**；若店长当前班次不在时间范围内，则下发给当前班次在范围内的**所有在班店员**。只要有一名员工提交，即视为该门店任务完成。

### 4.3 打卡（底部 Tab：打卡）

打卡功能以**月历视图**展示个人排班与打卡情况，是员工最高频使用的核心功能。

#### 4.3.1 打卡日历

日历中，**所有有排班的日期均直接展示班次名称**（如"早班"、"中班"），并以背景色和底部状态标识区分当天的打卡情况，让员工一眼看清排班安排与考勤结果。

**日历日期展示规则**

| 状态 | 日期格子内容 | 背景色 | 底部标识颜色 | 说明 |
|------|------------|--------|:----------:|------|
| 已排班（未来） | 日期数字 + 班次名称 | 浅蓝 | 蓝色 | 有排班计划，尚未到打卡日，不可点击打卡 |
| 已排班（今天/历史）+ 正常 | 日期数字 + 班次名称 | 浅绿 | 绿色 | 上下班均已正常打卡 |
| 已排班 + 迟到/早退 | 日期数字 + 班次名称 | 浅黄 | 黄色 | 有打卡但存在迟到或早退异常 |
| 已排班 + 缺勤/未打卡 | 日期数字 + 班次名称 | 浅红 | 红色 | 有排班但无任何打卡记录 |
| 无排班 | 日期数字（仅） | 透明 | 无 | 当天无排班计划，不可点击 |

**班次名称展示效果示意（以7月为例）：**

```
 日   一   二   三   四   五   六
                1    2    3    4    5
              早班  早班  中班  中班  中班
               6    7    8    9   10   11   12
             早班  休息  早班  早班  中班  中班  中班
```

> - 班次名称显示为班次的简称（取班次类型名称，如：早班、中班、晚班），字号较小显示于日期数字下方
> - 点击有排班的今天或历史日期，进入该日打卡界面；未来日期即使有排班也不可点击操作

日历顶部汇总当月统计：正常次数、迟到次数、早退次数、缺勤次数。

#### 4.3.2 打卡操作步骤

1. 在日历中点击**今天或过去某天**（未来日期不可操作）
2. 进入打卡界面，显示当天排班的班次信息（班次名称、上班时间、下班时间）
3. 系统根据当前时间与班次时间的关系，以及当天是否已有上班打卡记录，动态判断本次打卡类型，并在**开始拍照按钮**上显示对应文字（详见 4.3.4）
4. 界面中央**拍照图标**采用动态闪烁动画，提示用户点击；点击拍照图标或 **「开始拍照」** 按钮，调起手机前置摄像头
5. 在摄像头画面中，点击 **「拍照」** 按钮完成人脸拍摄
6. 预览照片确认无误后，点击 **「确认打卡」**
7. 系统自动获取当前 GPS 位置，验证是否在门店 100 米范围内：
   - **验证通过**：记录打卡成功，显示打卡时间
   - **验证失败**：提示距离门店 XX 米，超出 100 米范围，无法打卡
8. 打卡完成后，日历中该日期的状态自动更新

#### 4.3.3 迟到/早退判定规则

| 判定类型 | 规则 | 系统提示 |
|----------|------|----------|
| **迟到** | 上班打卡时间 > 班次开始时间 | 弹窗提示迟到，确认后记录为【迟到打卡】 |
| **早退** | 下班打卡时间 < 班次结束时间 | 弹窗提示早退，确认后记录为【早退打卡】 |
| **迟到早退** | 同一天既迟到又早退 | 综合记录，状态显示为【迟到早退】 |

#### 4.3.4 打卡界面与按钮动态逻辑

**拍照图标**：界面中央的拍照图标采用动态闪烁动画，吸引用户点击进入拍照流程。

**开始拍照按钮文字**：根据「当前时间」与「班次时间」的关系及「当天上班卡是否已打」动态切换，规则如下：

| 优先级 | 条件 | 按钮文字 | 页面提示 | 说明 |
|--------|------|----------|----------|------|
| 1 | 未打上班卡 **且** 当前时间 ≤ 班次开始时间 | **上班打卡** | — | 正常提前或准时打卡 |
| 2 | 未打上班卡 **且** 班次开始时间 < 当前时间 < 班次开始时间 + 4 小时 | **迟到打卡** | 当前时间晚于班次开始时间 | 迟到打卡，记录为迟到 |
| 3 | 未打上班卡 **且** 当前时间 ≥ 班次开始时间 + 4 小时 | **下班打卡** | 上班打卡缺卡 | 超过容忍窗口未打上班卡，直接打下班卡，状态记为缺勤 |
| 4 | 已打上班卡 **且** 当前时间 < 班次结束时间 | **早退打卡** | 当前时间早于班次结束时间，将记录为早退 | 提前下班，记录为早退 |
| 5 | 已打上班卡 **且** 当前时间 ≥ 班次结束时间 | **下班打卡** | — | 正常或延迟下班打卡 |

> **示例**：班次为 07:00—15:00  
> - 06:50 进入 → 按钮「**上班打卡**」  
> - 09:03 进入（未打上班卡）→ 按钮「**迟到打卡**」，提示：当前时间晚于班次开始时间 07:00  
> - 13:00 进入（已打上班卡）→ 按钮「**早退打卡**」，提示：当前时间早于班次结束时间 15:00，将记录为早退  
> - 15:30 进入（已打上班卡）→ 按钮「**下班打卡**」  
> - 13:00 进入（**未**打上班卡，13:00 > 11:00）→ 按钮「**下班打卡**」，提示：**上班打卡缺卡**，确认后记录为缺勤

> **打卡限制**：
> - 距离门店超过 **100 米**，系统拒绝打卡并提示距离
> - 未开启手机定位权限，无法获取位置，打卡被拦截
> - 未允许摄像头权限，无法拍照，打卡被拦截

### 4.4 排班（底部 Tab：排班，仅店长可见完整功能）

排班 Tab 内分为两个子页面，通过顶部分段控件切换：**班次管理** 和 **人员排班**。

#### 4.6.1 班次管理

**功能定位**：班次是门店的工作时间单元。各门店店长可根据本店实际经营需求，在班次类型框架下灵活定义本店的具体班次（如"早班·南京路 07:00—15:00"）。班次创建后方可用于人员排班。

> **层级关系**：PC端管理员配置「班次类型」（早班/中班/晚班/大夜班）→ 移动端店长创建「班次」（具体门店+时间段）→ 对员工进行「人员排班」。

**页面结构**

- 顶部显示本店已创建的班次列表，每条班次以卡片形式展示
- 右上角「＋ 新建班次」按钮，点击进入新建表单
- 每张班次卡片右侧提供「编辑」和「删除」操作

**班次列表卡片字段**

| 显示内容 | 说明 |
|----------|------|
| 班次名称 | 如：早班·南京路 |
| 班次类型标签 | 如：早班（对应 PC 端班次类型，带颜色标签）|
| 时间范围 | 如：07:00 — 15:00 |
| 时长 | 自动计算，如：8小时 |
| 状态 | 启用 / 停用 |

**新建/编辑班次字段说明**

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 班次名称 | ✓ | 建议格式：类型·门店简称，如"早班·南京路"，便于识别 |
| 班次类型 | ✓ | 从 PC 端已配置的班次类型中选择（早班/中班/晚班/大夜班）|
| 开始时间 | ✓ | 班次上班时间，以 30 分钟为精度滚轮选择 |
| 结束时间 | ✓ | 班次下班时间，以 30 分钟为精度滚轮选择 |

**操作说明**

| 操作 | 说明 |
|------|------|
| 新建班次 | 点击右上角「＋ 新建班次」进入表单，填写后保存 |
| 编辑班次 | 点击卡片右侧「编辑」，修改班次信息后保存；已关联排班的班次时间变更将自动更新打卡验证逻辑 |
| 删除班次 | 点击「删除」，仅可删除**未关联任何排班**的班次，已使用班次需先清除排班记录才可删除 |
| 停用班次 | 停用后不影响历史排班数据，但不可再用于新排班 |

> **注意**：夜班跨零点（如 23:00—次日 07:00）系统自动处理，打卡时间判断基于当天日历日期的班次记录。

#### 4.6.2 人员排班

店长为本店员工指定班次和排班周期。需先在「班次管理」中创建好班次，再在此处为员工分配。

**排班字段说明**

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 选择人员 | ✓ | 多选，可选择本店员工和已批准借调至本店的人员 |
| 选择班次 | ✓ | 从本店已创建并启用的班次中选择 |
| 开始日期 | ✓ | 排班生效开始日期 |
| 结束日期 | ✓ | 排班生效结束日期 |

**操作说明**
- 一次排班操作可同时为多名员工设置相同班次和周期，减少重复操作
- 排班创建后，员工的打卡日历中对应日期自动标记为"已排班"并显示班次名称
- 左划排班记录可删除对应排班安排

### 4.5 借调（底部 Tab：借调）

借调功能用于处理员工跨店支援的场景，如门店活动、人员短缺等情况。

#### 4.6.1 借调流程说明

```
调入门店店长发起申请
        ↓
调出门店店长查看并审批
        ↓
    审批通过 → 借调生效
        ↓
借调期间员工在调入门店打卡计为工时
        ↓
借调结束日期后自动失效，员工回归原店
```

#### 4.6.2 发起借调申请

由**调入门店**（需要人员的门店）的店长发起申请。

**申请字段说明**

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 调出门店 | ✓ | 选择人员来源门店（不能选择本店） |
| 借调人员 | ✓ | 从所选调出门店的员工中选择（只能看到对方门店人员） |
| 借调开始日期 | ✓ | 借调生效日期 |
| 借调结束日期 | ✓ | 借调结束日期，结束后自动失效 |
| 借调原因 | — | 说明借调原因，便于对方门店审批决策 |

#### 4.6.3 审批借调申请

由**调出门店**（被借调人员所在门店）的店长审批。

- 在 **「待审批」** Tab 中查看所有需要本店审批的借调申请
- 点击 **「批准」** → 借调立即生效，系统自动触发消息推送（详见 4.4.3.1）
- 点击 **「拒绝」** → 可填写拒绝原因，申请方可见

##### 4.5.3.1 审批通过后的消息推送

借调审批通过的瞬间，系统同时向以下两方发送消息通知：

**① 推送给借调员工本人**

通知员工借调安排详情，消息模版如下：

> 您将于 **XXXX年XX月XX日** 借调到 **（店号）** 门店，班次为 **XXXX**，上班时间为 **开始时间—结束时间**，请按照正确门店上班，感谢您的付出！

消息变量说明：

| 变量 | 取值来源 |
|------|---------|
| XXXX年XX月XX日 | 借调申请中的借调开始日期 |
| （店号） | 调入门店的门店编号（如：SH001） |
| XXXX（班次） | 调入门店店长为该员工安排的班次名称；若审批时尚未排班，则显示"待排班，请关注" |
| 开始时间—结束时间 | 对应班次的上下班时间（如：07:00—15:00） |

**② 推送给借调发起人（调入门店店长）**

通知发起人申请已获批准，消息模版如下：

> 您发起的借调申请已审批通过！**（员工姓名）** 将于 **XXXX年XX月XX日** 至 **XXXX年XX月XX日** 借调至本店，请及时安排班次。

> **关于推送渠道**：v1.0 版本消息以系统内站内消息形式展示，员工下次登录时在消息中心查看。生产环境建议对接企业微信 / 钉钉 / 短信，实现实时推送。

#### 4.5.4 借调生效后的影响

| 影响项 | 说明 |
|--------|------|
| 消息通知 | 审批通过后立即向**借调员工**和**借调发起人**分别推送通知消息 |
| 打卡验证 | 借调员工打卡时，系统以**借调目标门店（调入门店）的经纬度**作为 GPS 距离验证基准 |
| 员工档案 | 借调员工的**所属门店字段不发生任何变更**，组织归属保持原店不动 |
| 借调数据 | 借调记录完整保存（调出门店、调入门店、借调人员、起止日期、审批状态、原因等），可随时查询历史 |
| 排班显示 | 调入门店店长可在排班时看到并选择该借调员工 |
| 打卡看板 | 借调期间，该员工同时出现在调入门店的打卡看板中，并显示【借调】标签 |
| 工时归属 | 借调期间的打卡记录关联调入门店，工时计入调入门店 |
| 借调结束 | 超过借调结束日期后，借调自动失效，打卡验证自动切回原店经纬度，无需人工操作 |

#### 4.5.5 Tab 说明

| Tab | 内容 |
|-----|------|
| **待审批** | 需要本店店长审批的借调申请（红色角标提示数量） |
| **我发起的** | 本店店长发起的全部借调申请及其状态 |
| **全部记录** | 与本店相关的所有借调历史记录 |

### 4.6 看板（底部 Tab：看板）

打卡看板面向店长和督导，展示本店所有人员（含借调人员）在指定时间段内的考勤汇总数据。

#### 4.6.1 看板顶部汇总

| 统计项 | 说明 |
|--------|------|
| 总打卡次数 | 所选时间段内全店的打卡总次数 |
| 正常次数 | 状态为"正常"的打卡次数 |
| 异常次数 | 包含迟到、早退、缺勤的打卡次数总和 |
| 员工数 | 当前门店人员总数（含借调） |
| **总计划工时（h）** | 所选时间段内全店所有排班的计划工时总和 |
| **总实际工时（h）** | 按工时计算规则累计的有效工时总和 |

默认展示**当月**数据，可通过日期选择器切换日期区间。工时计算规则与 PC 端 3.8 章节一致。

#### 4.6.2 看板数据表格

每行显示一名员工的以下信息：

| 列名 | 说明 |
|------|------|
| 姓名 | 员工姓名，借调员工显示【借调】标签 |
| 班次 | 员工当前排班的时间段（如 07:00-15:00） |
| 应打次数 | 所选时间段内排班日期的总天数 |
| 正常次数 | 状态为"正常"的打卡天数 |
| 异常次数 | 迟到 + 早退 + 缺勤的总天数，**数字可点击** |
| 异常原因 | 以标签形式列出：迟到X次、早退X次、缺勤X次 |

#### 4.6.3 异常记录下钻查看

**触发方式**：点击看板数据表格中任意员工行的**异常次数数字**（红色数字），弹出该员工的异常明细底部弹窗。

**弹窗内容：**

- 顶部：员工姓名 + 所属班次 + 日期区间
- 明细列表，每条记录包含：

| 字段 | 说明 |
|------|------|
| 日期 | 具体异常打卡日期（如 2026-02-05，周二）|
| 班次名称 | 该日排班的班次（如 早班·南京路）|
| 应上班时间 | 班次规定开始时间 |
| 实际上班时间 | 实际打卡时间，未打卡显示"—" |
| 异常类型 | 迟到 / 早退 / 缺勤 |
| 异常时长 | 迟到/早退的分钟数（如 迟到 12 分钟）|

- 底部：「关闭」按钮

### 4.7 我的（底部 Tab：我的）

个人信息页面，展示当前登录员工的基本信息：

- 员工姓名、工号、角色
- 所属门店名称、店号、门店状态、门店类型

同时提供 **「退出登录」** 按钮，切换账号。

### 4.8 任务管理

**功能入口**：九宫格首页 → 任务管理

员工/督导在移动端查看和处理门店任务。不同角色视角不同：

| 角色 | 页签内容 |
|------|----------|
| 店长 / 店员 | 待提交、待整改、已完成 |
| 督导 | 待审批、整改审批、已完成 |

#### 4.8.1 任务列表

页面顶部以页签切换任务状态，下方以卡片形式展示任务列表。

**页签说明（员工/店长视角）：**

| 页签 | 说明 |
|------|------|
| 待提交 | 已下发、未超过截止时间、尚未提交的任务 |
| 待整改 | 督导审核后发起整改、等待重新提交的任务 |
| 已完成 | 已提交且审核通过（或无需审批）的任务 |
| 已过期 | 超过截止时间仍未提交的任务，不可再操作，仅供查看记录 |

> **已过期规则**：当系统时间超过任务的反馈截止时间且员工/门店未提交任何内容，该任务状态自动变更为「已过期」，从「待提交」移至「已过期」页签，无法继续提交。

**督导视角页签：** 待审批 / 整改审批 / 已完成 / 已过期（门店未提交导致过期的任务）

**任务卡片信息：**

| 字段 | 说明 |
|------|------|
| 任务名称 | — |
| 任务类型 | 商品陈列 / 熟食制作 / 清洁卫生 / 员工形象 |
| 截止时间 | 距截止时间不足 2 小时时红色高亮提醒；已过期卡片置灰展示 |
| 执行进度 | 已完成项 / 总项数（如 2/4）；已过期任务显示"已过期"覆盖标签 |
| 状态标签 | 待提交 / 待整改 / 已完成 / 已过期 |

点击任务卡片进入**任务详情页**；已过期任务详情仅展示任务内容，不显示拍照和提交功能。

#### 4.8.2 任务详情页（员工视角）

展示该任务的所有执行项，员工逐项拍照完成。

**页面结构：**

- 顶部：任务名称、类型、截止时间、执行门店
- 执行项列表（每项独立卡片）：
  - 执行标准文字说明
  - **查看标准** 按钮：点击展开显示执行标准文字+标准参考图片（轮播）
  - 拍照区域：支持**多张拍摄**（调起相机，不允许从相册选取），每张照片自动标注拍摄时间和 GPS 位置
  - 完成状态标记：已拍照则显示缩略图，未拍照显示"待拍照"
- 底部：**提交任务** 按钮（所有项均拍照后可提交；有未拍照项时按钮置灰并提示）

> **说明**：拍照强制使用相机实拍，系统自动获取拍摄时间戳和地理坐标，防止造假。

#### 4.8.3 任务详情页（督导视角 - 审核）

督导查看门店提交的任务，进行打分和审核。

**页面结构：**

- 顶部：任务信息 + 提交人 + 提交时间
- 执行项列表：展示每项的员工拍照结果（含拍摄时间、位置）及执行标准对比
- 底部审核操作区：
  - **任务总得分**：输入分值，默认满分 10 分（可修改）
  - **审核意见**：文本输入，可选填
  - **是否发起整改**：开关
    - **否**：点击「审核通过」完成审核
    - **是**：填写整改要求后点击「发起整改」，系统将整改任务推送给原提交人；提交人完成整改后重新提交，督导再次审核（**只做一次整改，不多次往复**）
  - 已整改任务：显示「整改内容」+「整改照片」，督导审核后点击「最终通过」结束流程

## 六、数据区隔设计（国家与公司）

系统支持按**国家**和**公司**进行数据区隔，实现多租户、多区域部署场景下的数据隔离与权限控制。

### 6.1 层级结构

```
国家 (Country)
  └── 公司 (Company)
        └── 区域 (Region)
              └── 门店 (Store)
                    └── 员工、班次、排班、打卡、借调、交接班等
```

| 层级 | 说明 | 数据隔离作用 |
|------|------|-------------|
| **国家** | 顶层维度，如：中国、越南 | 公司归属国家，支持多国部署 |
| **公司** | 隶属于国家，如：Ohmee 中国、Ohmee 越南 | 所有业务数据（区域、门店、员工、班次、打卡、借调、交接班）均归属公司，严格隔离 |
| **区域** | 隶属于公司 | 门店与员工的归属维度 |
| **门店** | 隶属于区域 | 打卡、排班、借调等业务执行单元 |

### 6.2 数据归属规则

| 实体 | 归属字段 | 说明 |
|------|----------|------|
| 公司 | countryId | 公司所属国家 |
| 区域 | companyId | 区域所属公司 |
| 门店 | companyId, regionId | 门店所属公司与区域 |
| 员工 | companyId, storeId, regionId | 员工所属公司、门店、区域 |
| 班次类型 | companyId | 班次类型为公司级配置 |
| 借调记录 | companyId | 借调仅限同公司内门店之间 |
| 交接班记录 | companyId, storeId | 交接班归属门店及公司 |

### 6.3 使用场景

| 端 | 场景 | 数据过滤方式 |
|----|------|-------------|
| **PC 端** | 总部管理员 | 顶部提供「国家」「公司」下拉选择器，切换后所有页面（区域、门店、员工、班次类型、交接班查询等）仅展示所选公司的数据 |
| **移动端** | 员工登录 | 登录后自动带入员工所属公司（companyId）与国家（countryId），所有操作与数据查看均限定于本公司范围内 |
| **督导** | 门店切换 | 督导仅可切换至所辖门店，且所辖门店限定为其所属公司下的门店 |

### 6.4 扩展说明

- 工号（empNo）在同一公司内唯一，不同公司可重复
- 店号（storeNo）在同一公司内唯一
- 借调仅支持同公司内门店之间发起与审批
- 后续可扩展：超级管理员角色，支持跨国家/公司数据查看与配置

## 七、核心业务流程

### 5.1 日常打卡流程

```
员工开启系统
    ↓
查看打卡日历，确认今日排班
    ↓
点击今日日期进入打卡界面
    ↓
点击「开始拍照」→ 调起前置摄像头
    ↓
对准脸部拍照
    ↓
确认照片 → 系统自动获取GPS定位
    ↓
┌─────────────────────┐
│ 距离门店 ≤ 100米？  │
└─────────────────────┘
     是 ↓              否 ↓
   继续验证         提示超出范围
     ↓               打卡失败
是否迟到/早退？
     ↓
   弹窗提醒
     ↓
确认 → 保存打卡记录
     ↓
    打卡成功
```

### 5.2 新开门店配置流程

```
PC端：区域管理 → 新增/确认所属区域
    ↓
PC端：员工管理 → 新增督导和店长账号
    ↓
PC端：门店管理 → 新增门店（地图选点确认位置）
    ↓
PC端：门店管理 → 为门店配置督导和店长
    ↓
移动端（店长登录）：排班 → 创建班次（早/中/晚班）
    ↓
移动端（店长登录）：排班 → 为员工安排班次和周期
    ↓
门店开始运营，员工可正常打卡
```

### 5.3 借调完整流程

```
调入门店店长（移动端）
    → 借调 Tab → 我发起的 → 发起借调申请
    → 选择调出门店、借调人员、起止日期、借调原因
    → 提交申请（借调记录创建，状态：待审批）
         ↓
调出门店店长（移动端）
    → 借调 Tab → 待审批（红色角标提醒数量）
    → 查看申请详情
    → 批准（借调立即生效）/ 拒绝（填写拒绝原因）
         ↓（批准后，借调记录状态变为：已批准）
系统自动推送消息
    → 向借调员工推送：借调时间、调入门店店号、班次及上下班时间
    → 向借调发起人推送：申请已通过，提醒及时安排班次
         ↓
借调员工档案
    → 所属门店字段保持原店不变（组织归属不受影响）
    → 借调记录完整保存，随时可查询历史
         ↓
调入门店店长
    → 排班页面中可看到该借调员工（标注[借调]）
    → 为其安排班次和周期
         ↓（借调期间）
借调员工打卡
    → 系统检测当日是否有生效的借调记录
    → 有效借调 → 以调入门店经纬度验证 GPS 距离
    → 打卡记录关联调入门店，工时计入调入门店
    → 调入门店看板中显示该员工考勤
         ↓（借调结束日期到达）
借调自动失效（无需人工操作）
    → 打卡验证自动切回原店经纬度
    → 员工回归原店正常打卡
    → 历史借调记录完整保留，可追溯查询
```

## 八、数据说明

### 8.1 数据存储

系统采用 **Python 后端 + SQLite 数据库** 的部署方案，实现多端数据实时共享与持久化存储。

#### 技术架构

```
┌─────────────────────────────────────┐
│           客户端（浏览器）            │
│   PC端（React）   移动端（React）     │
└────────────┬────────────────────────┘
             │ HTTP / REST API
┌────────────▼────────────────────────┐
│       Python 后端服务                │
│   Flask / FastAPI + 静态文件托管     │
│   端口：8000（可配置）               │
└────────────┬────────────────────────┘
             │ SQL
┌────────────▼────────────────────────┐
│         SQLite 数据库                │
│   文件：ohmee.db（本地单文件）        │
└─────────────────────────────────────┘
```

#### 各层说明

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | React + Vite（打包为静态文件） | 由 Python 服务统一托管，无需单独部署前端服务器 |
| 后端 | Python 3.10+，Flask 或 FastAPI | 提供 REST API 接口，处理业务逻辑 |
| 数据库 | SQLite（单文件，`ohmee.db`） | 前期部署无需安装数据库服务，文件即数据库，迁移方便 |
| 部署环境 | Windows / Linux / macOS 均可 | 推荐使用 Linux 服务器，Python venv 虚拟环境隔离依赖 |

#### SQLite 主要数据表

| 表名 | 对应数据 |
|------|---------|
| `regions` | 区域 |
| `stores` | 门店 |
| `employees` | 员工 |
| `shift_types` | 班次类型 |
| `shifts` | 班次 |
| `schedules` | 排班 |
| `attendances` | 打卡记录 |
| `transfers` | 借调记录 |

> **升级路径**：SQLite 适合前期单机或小规模部署。业务量增大后，可将数据库无缝迁移至 **MySQL** 或 **PostgreSQL**，后端代码仅需修改数据库连接配置，无需改动业务逻辑。

### 7.3 常见问题

**Q：打卡时提示"距离门店 XXX 米，超出 100 米范围"怎么办？**  
A：请确认您在门店内部或门店附近，若门店地图选点位置有偏差，需要联系管理员在 PC 端重新选点。

**Q：手机摄像头无法打开？**  
A：请在手机浏览器设置中，找到该网站的权限设置，允许访问摄像头。iOS 用户需在"设置 → Safari → 网站 → 摄像头"中允许。

**Q：打卡日历中当天没有出现排班标记？**  
A：请联系本店店长确认是否已完成排班，排班需要店长在移动端排班页面设置完成后才会显示在打卡日历中。

**Q：借调申请提交后，对方迟迟没有审批？**  
A：借调需要调出门店的店长在移动端审批，请联系对方店长在借调页面的"待审批"Tab 中处理。

**Q：借调结束后，员工的所属门店会变成借调门店吗？**  
A：不会。所属门店是员工的组织属性，代表编制归属，借调只是临时行为。借调期间和结束后，员工档案中的所属门店始终保持原店不变，借调记录单独保存，两者互不影响。

**Q：如何区分"调店"和"借调"？**  
A：调店（永久）：员工正式调往新门店工作，需由管理员在 PC 端员工管理中修改"所属门店"字段。借调（临时）：员工临时支援其他门店，使用移动端借调功能，有明确的起止日期，到期自动失效，所属门店不变。

**Q：员工工号忘记了怎么查？**  
A：管理员可在 PC 端"员工管理"中通过姓名搜索查询工号；员工本人也可在移动端"我的"页面查看自己的工号信息。

**Q：服务启动后，手机能访问吗？**  
A：能。手机和服务器需在同一局域网内，或服务器有公网 IP / 域名。手机浏览器输入 `http://服务器IP:8000/mobile/login` 即可访问移动端。建议为服务器配置域名并开启 HTTPS，以便手机摄像头和 GPS 权限正常工作（部分浏览器在非 HTTPS 环境下限制摄像头和定位访问）。

**Q：SQLite 数据库文件在哪里，如何备份？**  
A：数据库文件默认为项目根目录下的 `ohmee.db`，直接复制该文件即完成备份。建议设置定时任务每日自动备份，保存至独立目录或云存储。

**Q：历史数据能导出吗？**  
A：支持。在 PC 端门店管理和员工管理页面可导出对应 Excel 文件；打卡数据可在 PC 端「数据报表 → 打卡数据导出」中按日期范围、门店、员工等多维度筛选后导出 Excel。

### 10.3 移动端：审核交接班（店长角色）
